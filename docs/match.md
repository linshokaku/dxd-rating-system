# 試合運営仕様

## 目的

Discord Bot 上で 3v3 対戦の試合進行を管理し、親決定、勝敗報告、仮決定結果の承認、結果確定、ペナルティ反映を正しく扱う。

## スコープ

- 対象ゲームは DragxDrive のみとする。
- 対象フォーマットは 3v3 対戦のみとする。
- マッチングキューで成立した 6 人 1 組の対戦を 1 試合として扱う。
- Bot は試合進行のうち、親決定、勝敗報告、仮決定結果の承認、結果確定、ペナルティ管理を担当する。
- 部屋 ID の報告、部屋立ての進行、証拠画像の提示、試合ミスや切断への対処は運用で扱う。

## Bot が管理する範囲

- マッチ成立後の親募集と親決定
- 親決定時刻を起点としたタイマー管理
- 試合参加者 6 人による勝敗報告の受付
- 勝敗報告の仮判定と承認期間の管理
- 勝敗結果と各プレイヤーの報告状態・承認状態の確定
- 勝敗誤報告ペナルティと勝敗無報告ペナルティの承認期間終了後の自動付与
- admin による結果上書きとペナルティ加算・減算
- 報告数が少なすぎる場合、片方のチームからしか報告がない場合、同票を解消できない場合の admin 通知

## Bot が管理しない範囲

- 親決定後の部屋 ID 報告
- 部屋立て遅延の自動判定
- 試合ミス、遅刻、切断の自動判定
- 画像証拠の収集と真偽判定
- 運用判断による即時の無効試合化

補足:

- 部屋立て遅延、試合ミス、遅刻、切断は運用上の違反として扱う。
- Bot はこれらを自動検出・自動強制しない。
- 必要な反映は admin がコマンドで行う。

## 運用上の違反種別

運用上、以下の違反種別を扱う。

- 部屋立て遅延
- 試合ミス
- 遅刻
- 切断
- 勝敗誤報告
- 勝敗無報告

補足:

- 自動でペナルティを適用するのは `勝敗誤報告` と `勝敗無報告` のみとする。
- それ以外の違反は運用で判断し、admin が手動で反映する。

## 試合フロー

### 1. マッチ成立

- マッチングキューで 6 人そろった時点でマッチを作成する。
- Bot はチーム A / チーム B の組み分けを通知する。

### 2. 親募集

- マッチ成立直後から、参加者は親に立候補できる。
- 親募集期間は 5 分とする。
- 複数人が立候補した場合は先着順で親を決定する。
- 5 分以内に立候補者が現れなかった場合は、Bot が 6 人の中からランダムに親を決定する。

### 3. 親決定

- 親が決まった時刻を `parent_decided_at` とする。
- 以降の試合タイマーはすべて `parent_decided_at` を基準に計測する。
- 親決定後の部屋立てと部屋 ID 共有は運用で行い、Bot では管理しない。

### 4. 試合進行

- 部屋立て、試合開始、ゲーム間インターバル、試合終了は運用で進行する。
- Bot は試合中の進行そのものは管理しない。
- ただし、勝敗報告の受付開始時刻、締切時刻、承認期間はタイマーで管理する。

### 5. 勝敗報告

- 勝敗報告は、試合参加者 6 人のみが行える。
- 勝敗報告は投票形式とし、UX 上の投票先は以下の 4 種類とする。
  - 勝ち
  - 負け
  - 引き分け
  - 無効試合
- プレイヤーは、自分視点で試合に勝ったか負けたかを報告する。
- Bot は集計時に、各プレイヤーの所属チームに応じて報告内容を内部結果へ正規化する。
  - チーム A のプレイヤーの `勝ち` は `チーム A の勝ち` として扱う
  - チーム A のプレイヤーの `負け` は `チーム B の勝ち` として扱う
  - チーム B のプレイヤーの `勝ち` は `チーム B の勝ち` として扱う
  - チーム B のプレイヤーの `負け` は `チーム A の勝ち` として扱う
  - `引き分け` と `無効試合` は両チーム共通の結果として扱う
- `無効試合` の報告だけは、`match_created_at` の時点から受け付ける。
- `勝ち`、`負け`、`引き分け` は通常どおり `report_open_at` 以後に受け付ける。
- 1 プレイヤーは 1 件の最新報告だけを持つ。
- 通常の報告受付期間中は、自分の報告を何度でも上書きできる。

### 6. 承認期間

- 次のいずれかを満たした時点で、勝敗結果を仮判定し、5 分間の承認期間に入る。
  - 親が決定済みであり、かつ 6 人全員が勝敗報告を行った
  - 親が決定済みであり、かつ勝敗報告の締切時刻に到達した
- 承認期間中は、勝敗報告の変更や新規報告は受け付けない。
- 承認期間中に操作できるのは、最新の仮判定に対して `誤報告` または `無報告` になっているプレイヤーによる、仮決定結果の承認のみとする。
- 対象プレイヤーが承認期間中に仮決定結果を承認した場合、そのプレイヤーは `勝敗誤報告` または `勝敗無報告` の自動ペナルティ付与対象から外れる。
- Bot は承認対象プレイヤーに対して、承認できない場合は証拠を提示したうえで admin へ連絡するよう通達する。
- 承認期間が終わった時点で、勝敗結果、各プレイヤーの報告状態、各プレイヤーの承認状態を確定する。

### 7. 結果確定

- 確定後は参加者による勝敗報告の変更や承認を受け付けない。
- 承認期間終了後、確定結果に基づいて、`incorrect` または `not_reported` のプレイヤーのうち、承認期間中に承認していないプレイヤーへ勝敗誤報告ペナルティまたは勝敗無報告ペナルティを自動反映する。
- 承認期間中に承認したプレイヤーについては、対応する勝敗誤報告ペナルティまたは勝敗無報告ペナルティを付与しない。
- 条件に応じて admin へ確認通知を送る。

## タイマー仕様

### 親募集締切

- `match_created_at + 5 分`
- この時点までに親が決まっていなければ、Bot がランダムに親を決定する。

### 勝敗報告受付開始

- `parent_decided_at + 7 分`
- 7 分は、2 試合で決着した場合の最短終了想定とする。
- `勝ち`、`負け`、`引き分け` はこの時刻より前は受け付けない。
- `無効試合` は例外として、`match_created_at` の時点からいつでも受け付ける。

### 勝敗報告締切

- `parent_decided_at + 27 分`
- 27 分の内訳は以下のとおりとする。
  - 部屋立て 5 分
  - マッチング待機時間 5 分
  - 3 分の試合 3 回
  - 1 分のインターバル 3 回
  - 勝敗報告待機時間 5 分

補足:

- 部屋立て 5 分は締切計算に含めるが、部屋 ID 自体は Bot で管理しない。

### 承認期間

- 開始条件:
  - 親が決定済みであり、かつ 6 人全員の勝敗報告がそろった時点
  - または親が決定済みであり、かつ `parent_decided_at + 27 分`
- 終了時刻:
  - `approval_started_at + 5 分`

## 勝敗判定ルール

### 通常の判定

- 各プレイヤーの最新報告を内部結果へ正規化して集計する。
- 最も票数の多い選択肢を、その時点の基本結果とする。

### 同票時の判定

最大票数の選択肢が複数ある場合は、次の順で仮判定または確定結果を決める。

1. 誰も報告していない場合は、無効試合を採用する。
2. 親が最新報告を行っており、その正規化後の報告先が同票候補の中に含まれていれば、親の報告を採用する。
3. それでも決まらない場合は、無効試合とし、admin への連絡を行う。

補足:

- 引き分けと無効試合も通常の投票対象に含める。

### 承認期間中の扱い

- 承認期間に入った後は、仮決定結果は固定し、参加者による勝敗報告の変更では更新しない。
- Bot は仮決定結果、対象プレイヤー、承認期限を承認対象プレイヤーへ通知する。
- 通知時には、承認できない場合は証拠を提示したうえで admin へ連絡する必要があることも案内する。
- 承認は報告内容の修正ではなく、仮決定結果への同意として扱う。

## 各プレイヤーの報告状態

結果確定時、各プレイヤーの報告状態を以下のいずれかで記録する。

- `correct`
  - 最新報告を正規化した結果が、確定結果と一致している
- `incorrect`
  - 最新報告を正規化した結果が、確定結果と一致しない
- `not_reported`
  - 承認期間開始時点までに報告を提出していない

補足:

- 承認の有無によって `report_status` は変更しない。
- `not_reported` のプレイヤーが承認期間中に承認した場合でも、表示上の報告状態は `not_reported` のままとする。

## 各プレイヤーの承認状態

承認期間の対象プレイヤーについて、以下のいずれかを記録する。

- `not_required`
  - 承認期間開始時点で `correct` であり、承認を必要としない
- `pending`
  - 承認期間開始時点で `incorrect` または `not_reported` であり、まだ承認していない
- `approved`
  - 承認期間中に仮決定結果を承認した
- `not_approved`
  - 承認期間終了時点までに承認しなかった

## admin 通知条件

以下のいずれかに該当する場合、Bot は admin に確認通知を送る。

- ここでいう「勝敗報告を行ったプレイヤー」は、承認期間開始時点で勝敗報告を提出しているプレイヤーを指す。
- 勝敗報告を行ったプレイヤーが 2 人以下である
- 勝敗報告を行ったプレイヤーが片方のチームにしか存在しない
- 同票が解消できず、仮決定結果を無効試合として扱う場合

通知タイミング:

- 承認期間開始時点の状態を基準に判定する。
- 通知は承認期間終了後の結果確定時に行う。
- admin 通知は結果確定のアナウンスの後に送る。

## ペナルティ仕様

### 自動適用するペナルティ

- 勝敗誤報告
  - 結果確定時に `incorrect` かつ `approved` ではないプレイヤーへ自動で `+1` する
- 勝敗無報告
  - 結果確定時に `not_reported` かつ `approved` ではないプレイヤーへ自動で `+1` する

補足:

- 自動ペナルティの付与判定は承認期間終了後に行う。
- 承認期間中に仮決定結果を承認したプレイヤーについては、対応する勝敗誤報告ペナルティまたは勝敗無報告ペナルティを付与しない。

### 自動適用しない違反

以下は Bot が自動判定・自動加算しない。

- 部屋立て遅延
- 試合ミス
- 遅刻
- 切断

これらは運用で判断し、admin が手動で加算・減算する。

### admin による手動補正

- 勝敗誤報告
- 勝敗無報告
- 部屋立て遅延
- 試合ミス
- 遅刻
- 切断

上記すべての違反種別について、admin は手動で `+1` と `-1` を行えるようにする。

## コマンド仕様

### 共通方針

- 将来的には UX を被せて利便性を向上させることを検討する。
- ただし、コマンドインターフェースの時点では、試合に紐づく操作を行うコマンドはすべて `match_id` を引数に取る。
- 試合に紐づく操作とは、少なくとも以下を指す。
  - 親に立候補する
  - 勝敗報告を行う
  - 仮決定結果を承認する
  - admin による試合結果上書きを行う
- 開発用のダミーユーザー操作コマンドでは、引数の順序は `dummy_user_id`, `match_id` とする。
- admin が操作するコマンドは開発用コマンドではないため、`dev` prefix ではなく `admin` prefix を用いる。

### ユーザー向けコマンド

最低限、以下を想定する。

- 親に立候補するコマンド
- 勝ちを報告するコマンド
- 負けを報告するコマンド
- 引き分けを報告するコマンド
- 無効試合を報告するコマンド
- 仮決定結果を承認するコマンド

#### 親に立候補するコマンド

- 実行者は試合参加者のみ。
- 引数として `match_id` を取る。
- 親募集期間中のみ受け付ける。
- すでに親が決まっている場合は失敗する。
- 複数人が立候補した場合は、最初に受理されたものを採用する。

#### 勝敗報告コマンド

- 実行者は試合参加者のみ。
- 引数として `match_id` を取る。
- 勝敗報告は、以下の 4 つの別コマンドとして提供する。
  - 勝ちを報告するコマンド
  - 負けを報告するコマンド
  - 引き分けを報告するコマンド
  - 無効試合を報告するコマンド
- 各コマンドは、それぞれ対応する結果を報告する。
- 入力はプレイヤー自身の視点で解釈する。
- Bot は実行されたコマンドの種類と所属チームに応じて、結果を `チーム A の勝ち`、`チーム B の勝ち`、`引き分け`、`無効試合` のいずれかへ正規化して保存または集計する。
- 勝ち、負け、引き分けを報告する各コマンドは `report_open_at` より前は失敗する。
- 無効試合を報告するコマンドは例外として `match_created_at` の時点から受け付ける。
- 通常の報告受付期間中は、全参加者が何度でも上書きできる。
- 承認期間中は受け付けない。
- 結果確定後は失敗する。

#### 仮決定結果を承認するコマンド

- 実行者は試合参加者のみ。
- 引数として `match_id` を取る。
- 承認期間中のみ受け付ける。
- 最新の仮判定に対して `incorrect` または `not_reported` になっているプレイヤーのみ実行できる。
- このコマンドは、仮決定結果に同意する意思表示として扱い、勝敗報告そのものを修正するものではない。
- 承認が成功した場合、そのプレイヤーは `勝敗誤報告` または `勝敗無報告` の自動ペナルティ付与対象から外れる。
- `not_reported` のプレイヤーが承認した場合でも、報告状態は `not_reported` のままとする。
- 仮決定結果を承認できない場合、Bot は証拠を提示したうえで admin へ連絡するよう対象プレイヤーに案内する。

補足:

- 部屋 ID 報告コマンドは持たない。

### admin 向けコマンド

最低限、以下を想定する。

- 試合結果を上書きするコマンド
- 各違反種別のペナルティを `+1` するコマンド
- 各違反種別のペナルティを `-1` するコマンド

補足:

- admin 向けコマンドの prefix は `admin` とする。

#### 試合結果上書きコマンド

- admin のみ実行できる。
- 引数として `match_id` を取る。
- 証拠画像や運用判断に基づいて、試合結果を上書きできる。
- 上書き時は、必要に応じて各プレイヤーの報告状態と自動ペナルティも再計算または補正できるようにする。

#### ペナルティ加算・減算コマンド

- admin のみ実行できる。
- 加算コマンドと減算コマンドは分ける。
- 引数は対象プレイヤーの `user_id` のみとする。
- コマンドは対象違反のカウントを単純に `+1` または `-1` するだけの挙動とする。

対象違反種別:

- 部屋立て遅延
- 試合ミス
- 遅刻
- 切断
- 勝敗誤報告
- 勝敗無報告

### 開発用ダミーユーザー操作コマンド

- 開発用のダミーユーザー操作コマンドでは、試合に紐づく操作に `dummy_user_id` と `match_id` を渡す。
- 引数順は必ず `dummy_user_id`, `match_id` とする。
- ダミーユーザーによる勝敗報告も、通常ユーザー向けと同様に結果別の別コマンドとして扱う。
- 例:
  - ダミーユーザーを親に立候補させるコマンド
  - ダミーユーザーに勝ちを報告させるコマンド
  - ダミーユーザーに負けを報告させるコマンド
  - ダミーユーザーに引き分けを報告させるコマンド
  - ダミーユーザーに無効試合を報告させるコマンド
  - ダミーユーザーに仮決定結果を承認させるコマンド

## 運用ルール

### 部屋立て遅延

- 親決定後 5 分以内に部屋立てが進まない場合は、運用上の違反として扱う。
- 運用上は即無効試合として扱う想定とする。
- Bot は自動検出しない。
- 必要なら admin が試合結果上書きとペナルティ反映を行う。

### 試合ミス

- 親が試合の組み方を間違えた場合は、運用上の違反として扱う。
- 運用上は、その試合が終わった時点で部屋を解散し、無効試合として扱う想定とする。
- Bot は自動で無効試合化しない。
- 必要なら admin が結果上書きやペナルティ反映を行う。

### 遅刻

- 運用上の違反として扱う。
- 運用上は即無効試合として扱う想定とする。
- Bot は自動検出しない。
- 必要なら admin が試合結果上書きとペナルティ反映を行う。

### 切断

- 運用上の違反として扱う。
- 運用上は、その試合が終わった時点で部屋を解散し、無効試合として扱う想定とする。
- Bot は自動で無効試合化しない。
- 必要なら admin が結果上書きやペナルティ反映を行う。

### 証拠と異議申し立て

- 部屋立て遅延は、ログまたはコメントをもとに運用判断する。
- 試合ミスは、撮影時刻付きのリザルトのスクリーンショットを証拠候補とする。
- 遅刻は、撮影時刻付きのメンバータブのスクリーンショットを証拠候補とする。
- 切断は、撮影時刻付きの試合後リザルトのスクリーンショットを証拠候補とする。
- 無効試合に対する異議申し立ては、証拠提示と admin 判断で扱う。

## 必要な管理データ

### 試合単位

- `match_id`
- `created_at`
- `team_a_player_ids`
- `team_b_player_ids`
- `parent_player_id`
- `parent_decided_at`
- `report_open_at`
- `report_deadline_at`
- `approval_started_at`
- `approval_deadline_at`

## 状態遷移

最低限、以下の状態を持つ。

- `waiting_for_parent`
  - 親募集期間中
- `waiting_for_result_reports`
  - 親決定後、勝敗報告受付中
- `awaiting_result_approvals`
  - 承認期間中
- `finalized`
  - 結果確定済み

## まとめ

- 親決定までは Bot が進行を管理する。
- 部屋立てと試合進行そのものは運用で扱う。
- 勝敗報告は 6 人の自己視点の投票として受け付け、内部ではチーム結果へ正規化して扱う。
- 仮判定後に 5 分の承認期間を設け、対象プレイヤーは仮決定結果を承認できる。
- 自動ペナルティは勝敗誤報告と勝敗無報告のみとする。
- それ以外の違反は admin がコマンドで反映する。
