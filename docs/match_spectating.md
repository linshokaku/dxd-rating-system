# 試合観戦応募仕様（素案）

## 目的

Discord Bot 上で、運営中の試合に対する観戦応募を受け付け、試合ごとの観戦枠上限を正しく管理する。

本仕様で Bot が扱うのは、**観戦枠の応募受付と人数管理**までとする。  
実際の部屋招待、入室確認、観戦開始後の進行管理は運用で扱う。

## スコープ

本仕様では、少なくとも以下を扱う。

- ユーザーが `match_id` を指定して観戦応募できること
- 1 試合ごとの観戦枠上限を Bot が管理できること
- 同一ユーザーの重複応募を防ぐこと
- 試合参加者自身による観戦応募を防ぐこと
- 観戦受付終了後の試合に対する応募を防ぐこと

本仕様では、以下は扱わない。

- 実際の room 招待
- 観戦応募時の追加通知
- 観戦者一覧の表示コマンド
- 観戦辞退コマンド
- admin による観戦者の強制追加・強制削除
- ユーザーごとの観戦権限停止ルールの詳細

## 用語

### 観戦応募

観戦希望者が、ある試合の観戦枠を 1 つ確保する操作を指す。  
空き枠がある場合は即時に成立し、承認待ち状態は設けない。

### 観戦枠上限

1 試合に対して確保できる観戦者数の上限を指す。  
観戦枠上限は、次の式で求める。

`max(0, 12 - 試合参加人数)`

ここでいう `試合参加人数` は、その `match_id` に紐づく `match_participants` の件数とする。  
現行の 3v3 試合では参加人数は `6` となるため、観戦枠上限は `6` となる。

### 観戦受付中の試合

本仕様では、以下の状態にある試合を観戦受付中とする。

- `waiting_for_parent`
- `waiting_for_result_reports`

以下の状態は観戦受付対象外とする。

- `awaiting_result_approvals`
- `finalized`

補足:

- `awaiting_result_approvals` に入った時点では、試合プレイ自体は終了している前提で扱う。

## 基本方針

- コマンド名は暫定で `/match_spectate` とする
- 観戦応募は先着順で受け付ける
- 初期版では、`/register` 済みのユーザーのみ観戦応募できる
- 試合参加者は、自分が参加している試合に対して観戦応募できない
- 同一ユーザーは、同一試合に対して 1 件まで観戦応募できる
- 観戦応募が成立したユーザーは、その試合の観戦枠を 1 つ占有する
- 観戦枠は、少なくとも試合の結果確定時に解放される
- 観戦応募時の専用通知は送らない

補足:

- 初期版では、別試合への同時応募は制限しない

## `/match_spectate`

### 目的

実行者を、指定した試合の観戦者として登録する。

### 実行者

`/register` 済みの Discord ユーザー。

### 入力

- `match_id` (必須): 観戦応募したい試合の `match_id`

### 成功条件

- 実行者がプレイヤー登録済みである
- 指定した `match_id` の試合が存在する
- 指定した試合が観戦受付中である
- 実行者がその試合の参加者ではない
- 実行者がその試合へ未応募である
- 現在の観戦者数が観戦枠上限未満である

### 処理

1. トランザクションを開始する
2. 対象試合に対して、試合系の他操作と同じ単位で排他ロックを取得する
3. `match_id` に対応する試合と現在の active state を取得する
4. 試合が観戦受付中でなければ失敗する
5. `match_participants` を取得し、参加人数を数える
6. 観戦枠上限を `max(0, 12 - 試合参加人数)` で計算する
7. 実行者が試合参加者に含まれていれば失敗する
8. 実行者の既存の観戦応募が同一試合に存在すれば失敗する
9. 現在の有効な観戦応募数を数える
10. 観戦応募数が上限以上なら失敗する
11. 新しい観戦応募行を作成する
12. commit する

### 正常時の応答

- 応答例: `観戦応募を受け付けました。現在 3 / 6 人です。`

補足:

- コマンド応答は通常の slash command 応答として channel に表示する
- 観戦応募専用の非同期通知は送らない

### エラー時の応答

- プレイヤー未登録の場合:
  - `プレイヤー登録が必要です。先に /register を実行してください。`
- 指定した試合が存在しない場合:
  - `指定した試合が見つかりません。`
- 指定した試合が観戦受付中でない場合:
  - `この試合は観戦受付を終了しています。`
- 実行者がその試合の参加者である場合:
  - `この試合の参加者は観戦応募できません。`
- すでに応募済みの場合:
  - `すでにこの試合へ観戦応募済みです。`
- 観戦枠が埋まっている場合:
  - `この試合の観戦枠は埋まっています。`
- 内部エラーが発生した場合:
  - `観戦応募に失敗しました。管理者に確認してください。`
  - 内部例外の詳細は Discord に出さない

## データモデル案

観戦応募は、`match_spectators` のような専用テーブルで管理する想定とする。

最低限必要なカラム:

- `id`
- `match_id`
- `player_id`
- `status`
- `created_at`
- `removed_at`
- `removal_reason`

`status` は少なくとも以下を持つ。

- `active`
- `closed`

補足:

- 初期版では `active -> closed` の遷移理由は結果確定のみとする
- `player_id` は既存のプレイヤー登録データを参照する
- `closed` は、結果確定などにより観戦枠として有効でなくなった状態を表す

制約:

- `status = 'active'` の行は、同一 `(match_id, player_id)` に対して 1 件まで
- `match_id` と `status` に index を張り、試合ごとの観戦者数を数えやすくする

## 状態遷移

- `なし -> active`
  - `/match_spectate`
- `active -> closed`
  - 試合結果確定

## 整合性と排他

- 観戦応募数の上限超過を防ぐため、応募処理は試合単位で直列化する
- 既存の試合操作と同様に、`match_id` 単位の advisory lock を使う方針が自然である
- 試合結果確定と観戦応募が競合した場合は、先にロックを取得した側の結果を正として扱う
- そのため、結果確定が先に反映された場合、後続の観戦応募は `観戦受付終了` として失敗する

## 権限制御との関係

- 将来的に、キュー参加権限などと同列のユーザー権限制御の一部として、特定ユーザーの観戦応募権限を停止できるようにする予定がある
- ただし、その権限制御の管理コマンド、判定方法、永続化方式は本仕様では扱わない
