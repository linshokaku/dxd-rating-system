# 試合観戦応募仕様

## 目的

運営中の試合に対する観戦応募を受け付け、試合ごとの観戦枠上限を正しく管理する。

本仕様で Bot が扱うのは、観戦枠の応募受付と人数管理までとする。  
実際の部屋招待、入室確認、観戦開始後の進行管理は運用で扱う。

## スコープ

本仕様では少なくとも以下を扱う。

- ユーザーが `match_id` を指定して観戦応募できること
- 1 試合ごとの観戦枠上限を Bot が管理できること
- 同一ユーザーの重複応募を防ぐこと
- 試合参加者自身による観戦応募を防ぐこと
- 観戦受付終了後の試合に対する応募を防ぐこと

## 観戦枠上限

観戦枠上限は以下で求める。

```text
max(0, 12 - 試合参加人数)
```

ここでいう試合参加人数は、その `match_id` に紐づく `match_participants` の件数とする。

フォーマット別の上限:

- `1v1`: `12 - 2 = 10`
- `2v2`: `12 - 4 = 8`
- `3v3`: `12 - 6 = 6`

## 観戦受付中の試合

以下の状態にある試合を観戦受付中とする。

- `waiting_for_parent`
- `waiting_for_result_reports`

以下の状態は観戦受付対象外とする。

- `awaiting_result_approvals`
- `finalized`

## 基本方針

- コマンド名は暫定で `/match_spectate` とする
- 観戦応募は先着順で受け付ける
- 初期版では `/register` 済みのユーザーのみ観戦応募できる
- 試合参加者は、自分が参加している試合に対して観戦応募できない
- 同一ユーザーは、同一試合に対して 1 件まで観戦応募できる
- 観戦枠は、少なくとも試合の結果確定時に解放される

## `/match_spectate`

### 入力

- `match_id`

### 成功条件

- 実行者がプレイヤー登録済みである
- 指定した `match_id` の試合が存在する
- 指定した試合が観戦受付中である
- 実行者がその試合の参加者ではない
- 実行者がその試合へ未応募である
- 現在の観戦者数が観戦枠上限未満である

### 処理

1. トランザクションを開始する
2. `match_id` 単位で排他ロックを取得する
3. 対象試合と現在の active state を取得する
4. 試合が観戦受付中でなければ失敗する
5. `match_participants` を取得し、参加人数を数える
6. 観戦枠上限を `max(0, 12 - 試合参加人数)` で計算する
7. 実行者が試合参加者に含まれていれば失敗する
8. 実行者の既存の観戦応募が同一試合に存在すれば失敗する
9. 現在の有効な観戦応募数を数える
10. 観戦応募数が上限以上なら失敗する
11. 新しい観戦応募行を作成する
12. commit する

### 正常時の応答

- `観戦応募を受け付けました。現在 3 / 8 人です。`

## データモデル案

観戦応募は `match_spectators` のような専用テーブルで管理する。

最低限必要なカラム:

- `id`
- `match_id`
- `player_id`
- `status`
- `created_at`
- `removed_at`
- `removal_reason`

`status` は少なくとも以下を持つ。

- `active`
- `closed`

制約:

- `status = 'active'` の行は、同一 `(match_id, player_id)` に対して 1 件まで
- `match_id` と `status` に index を張り、試合ごとの観戦者数を数えやすくする

## 状態遷移

- `なし -> active`
  - `/match_spectate`
- `active -> closed`
  - 試合結果確定
