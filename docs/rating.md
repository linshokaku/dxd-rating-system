# 3vs3 レート計算方式仕様

## 目的

Discord コミュニティ向けの 3vs3 対戦において、各プレイヤーに **一軸のレート** を持たせ、以下を実現する。

* 対戦前に 2 チームの **期待勝率** を計算できる
* 対戦後に各プレイヤーの **レートを更新** できる
* 実装が比較的簡単で、挙動を説明しやすい
* 新規プレイヤーは早く収束し、古参プレイヤーは安定する

---

## 基本方針

各プレイヤーのレート `r` を、そのまま足し合わせるのではなく、まず強さ指標 `q` に変換する。

[
q_i = 10^{r_i / 400}
]

チーム強さは、メンバーの `q` の和とする。

[
Q_A = \sum_{i \in A} q_i,\qquad
Q_B = \sum_{j \in B} q_j
]

この `Q` を用いて、チーム A の期待勝率を求める。

[
p_A = \frac{Q_A}{Q_A + Q_B}
]

[
p_B = \frac{Q_B}{Q_A + Q_B} = 1 - p_A
]

この方式は、Elo の `Q = 10^(R/400)` 表現をチーム戦に拡張したものとみなす。

---

## 用語定義

### プレイヤーごとの保持値

各プレイヤーは少なくとも以下を保持する。

* `rating`
  現在のレート。実数で保持してよい
* `games_played`
  通算レート対象試合数

必要に応じて以下も保持可能。

* `wins`
* `losses`
* `draws`
* `last_played_at`

---

## 初期値

新規プレイヤーの初期値は以下とする。

* `rating = 1500`
* `games_played = 0`

`1500` は基準点であり、相対比較に使う。

---

## 期待勝率の計算

### 入力

* チーム A の 3 人のレート
  `r_A1, r_A2, r_A3`
* チーム B の 3 人のレート
  `r_B1, r_B2, r_B3`

### 手順

各プレイヤーについて `q` を計算する。

[
q_i = 10^{r_i / 400}
]

チーム強さを合計する。

[
Q_A = q_{A1} + q_{A2} + q_{A3}
]

[
Q_B = q_{B1} + q_{B2} + q_{B3}
]

期待勝率を計算する。

[
p_A = \frac{Q_A}{Q_A + Q_B}
]

[
p_B = \frac{Q_B}{Q_A + Q_B}
]

### 備考

* `p_A + p_B = 1`
* 3vs3 以外に拡張する場合も、同人数チームなら同様に `q` の総和で扱える
* マッチング時は `p_A` が 0.5 に近い組み合わせほど公平とみなせる

---

## 試合結果の表現

チーム A 視点の実結果を `y` とする。

* A 勝ち: `y = 1`
* 引き分け: `y = 0.5`
* B 勝ち: `y = 0`

本仕様では、**引き分けを正式に採用する**。

---

## レート更新式

### 基本式

チーム A のプレイヤー `i` の更新式:

[
r_i' = r_i + K_i \cdot \frac{q_i}{Q_A} \cdot (y - p_A)
]

チーム B のプレイヤー `j` の更新式:

[
r_j' = r_j - K_j \cdot \frac{q_j}{Q_B} \cdot (y - p_A)
]

ここで、

* `q_i = 10^(r_i/400)`
* `Q_A = Σ q_i`
* `Q_B = Σ q_j`
* `p_A = Q_A / (Q_A + Q_B)`

### 直感

* `(y - p_A)` は「実結果 − 期待結果」
* 予想より良い結果ならプラス
* 予想より悪い結果ならマイナス
* チーム内では `q_i / Q_team` が大きい人ほど更新量が大きい

つまり、試合前により強いと見なされていたプレイヤーほど、その試合結果への責任も大きくなる。

---

## K の設計

### 方針

K は固定値ではなく、**通算試合数に応じて段階的に変える**。

狙いは以下。

* 新規プレイヤーは早く適正レートへ寄せる
* 試合数の多いプレイヤーは安定させる
* 実装を簡潔に保つ

### 採用案

各プレイヤーの `games_played` に応じて K を以下で決定する。

[
K_i =
\begin{cases}
40 & \text{if } games_played < 20 \
32 & \text{if } 20 \le games_played < 100 \
24 & \text{if } games_played \ge 100
\end{cases}
]

### 備考

* これはプレイヤーごとに個別適用する
* 同じ試合でも、各プレイヤーの K は異なってよい
* より安定寄りにしたい場合は `32/24/16` などへ調整可能

---

## 試合処理手順

1 試合ごとに以下を実行する。

### 1. 対戦前データ取得

* チーム A の 3 人の `rating`, `games_played`
* チーム B の 3 人の `rating`, `games_played`

### 2. 各プレイヤーの q を計算

[
q_i = 10^{r_i / 400}
]

### 3. チーム強さを計算

[
Q_A = \sum_{i\in A} q_i,\qquad
Q_B = \sum_{j\in B} q_j
]

### 4. 期待勝率を計算

[
p_A = \frac{Q_A}{Q_A + Q_B}
]

### 5. 試合結果 y を決定

* A 勝ちなら `y = 1`
* 引き分けなら `y = 0.5`
* B 勝ちなら `y = 0`

### 6. 各プレイヤーの K を決定

`games_played` に応じて K を割り当てる。

### 7. 各プレイヤーの新レートを計算

A 側:

[
r_i' = r_i + K_i \cdot \frac{q_i}{Q_A} \cdot (y - p_A)
]

B 側:

[
r_j' = r_j - K_j \cdot \frac{q_j}{Q_B} \cdot (y - p_A)
]

### 8. 保存

* `rating = r'`
* `games_played += 1`

---

## 実装上の注意

### レートは整数でなくてもよい

内部計算は浮動小数で持ち、表示時だけ丸めるのがよい。

例:

* DB保存: `1523.4187`
* 表示: `1523`

### q の再利用

同一試合内では、更新前レートから計算した `q` を使い回す。
更新途中で `q` を再計算しない。

### 同時更新する

6人全員の更新量を先に計算し、その後まとめて反映する。
片側だけ先に更新してから相手側を計算しない。

### 引き分けの扱い

引き分けは `y = 0.5` として処理する。
更新式自体は勝敗時と同一であり、結果値 `y` のみが異なる。

---

## この方式の利点

* 一軸レートで 3vs3 の期待勝率を自然に計算できる
* 実装が比較的簡単
* Elo に近い説明が可能
* 新規と古参で更新速度を分けられる
* チーム内の強いプレイヤーほど更新寄与が大きい

---

## この方式の限界

* ロール相性を表現できない
* シナジーを表現できない
* 特定の組み合わせだけ強い、という効果を表せない
* 3人全員の寄与を独立加算的に扱っている

そのため、編成相性が非常に重要なゲームでは限界がある。
ただし Discord コミュニティ向けの初期実装としては十分現実的。
