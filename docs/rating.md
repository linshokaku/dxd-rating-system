# 3vs3 レート計算方式仕様

## 目的

Discord コミュニティ向けの 3vs3 対戦において、各プレイヤーに **一軸のレート** を持たせ、以下を実現する。

- 対戦前に 2 チームの **期待勝率** を計算できる
- 対戦後に各プレイヤーの **レートを更新** できる
- 実装が比較的簡単で、挙動を説明しやすい
- 新規プレイヤーは早く収束し、古参プレイヤーは安定する

本仕様は、1 試合単位のレート計算式と、その計算に必要な永続化前提を定める。  
admin による試合結果修正時のレーティング再計算は、別途 `match_result_correction.md`
に従うものとする。

---

## 基本方針

各プレイヤーのレート `r` を、そのまま足し合わせるのではなく、まず強さ指標 `q` に変換する。

```text
q_i = 10^(r_i / 400)
````

チーム強さは、メンバーの `q` の和とする。

```text
Q_A = sum(q_i for i in A)
Q_B = sum(q_j for j in B)
```

この `Q` を用いて、チーム A の期待勝率を求める。

```text
p_A = Q_A / (Q_A + Q_B)
p_B = Q_B / (Q_A + Q_B) = 1 - p_A
```

この方式は、Elo の `Q = 10^(R/400)` 表現をチーム戦に拡張したものとみなす。

---

## 用語定義

### プレイヤーごとの保持値

各プレイヤーは少なくとも以下を保持する。

* `rating`
  現在のレート。実数で保持してよい
* `games_played`
  通算レート対象試合数
* `wins`
* `losses`
* `draws`

必要に応じて以下も保持してよい。

* `last_played_at`

### 試合ごとの保持値

将来の試合結果修正とレーティング再計算に備え、レート対象となりうる試合ごとに
少なくとも以下を永続化する。

* `rated_at`
  レーティング順序を決める不変の時刻
* `final_result`
* `team_a_player_ids`
* `team_b_player_ids`
* 各参加プレイヤーの `rating_before`
* 各参加プレイヤーの `games_played_before`
* 各参加プレイヤーの `wins_before`
* 各参加プレイヤーの `losses_before`
* 各参加プレイヤーの `draws_before`

`rated_at` は admin による結果修正後も変更しない。  
結果修正時の再計算方式は `match_result_correction.md` で定義する。

---

## 初期値

新規プレイヤーの初期値は以下とする。

* `rating = 1500`
* `games_played = 0`

`1500` は基準点であり、相対比較に使う。

---

## 期待勝率の計算

### 入力

* チーム A の 3 人のレート
  `r_A1, r_A2, r_A3`
* チーム B の 3 人のレート
  `r_B1, r_B2, r_B3`

### 手順

各プレイヤーについて `q` を計算する。

```text
q_i = 10^(r_i / 400)
```

チーム強さを合計する。

```text
Q_A = q_A1 + q_A2 + q_A3
Q_B = q_B1 + q_B2 + q_B3
```

期待勝率を計算する。

```text
p_A = Q_A / (Q_A + Q_B)
p_B = Q_B / (Q_A + Q_B)
```

### 備考

* `p_A + p_B = 1`
* 3vs3 以外に拡張する場合も、同人数チームなら同様に `q` の総和で扱える
* マッチング時は `p_A` が `0.5` に近い組み合わせほど公平とみなせる

---

## 試合結果の表現

チーム A 視点の実結果を `y` とする。

* A 勝ち: `y = 1`
* 引き分け: `y = 0.5`
* B 勝ち: `y = 0`
* 無効試合: レート計算対象外

本仕様では、**引き分けを正式に採用する**。  
無効試合は試合記録としては保持するが、レーティング計算上は
試合が行われなかったものとして扱う。

---

## レート更新式

### 基本式

チーム A のプレイヤー `i` の更新式:

```text
r_i' = r_i + K_i * (q_i / Q_A) * (y - p_A)
```

チーム B のプレイヤー `j` の更新式:

```text
r_j' = r_j - K_j * (q_j / Q_B) * (y - p_A)
```

ここで、

* `q_i = 10^(r_i / 400)`
* `Q_A = sum(q_i for i in A)`
* `Q_B = sum(q_j for j in B)`
* `p_A = Q_A / (Q_A + Q_B)`

### 直感

* `(y - p_A)` は「実結果 - 期待結果」
* 予想より良い結果ならプラス
* 予想より悪い結果ならマイナス
* チーム内では `q_i / Q_team` が大きい人ほど更新量が大きい

つまり、試合前により強いと見なされていたプレイヤーほど、その試合結果への責任も大きくなる。

---

## K の設計

### 方針

K は固定値ではなく、**通算試合数に応じて段階的に変える**。

狙いは以下。

* 新規プレイヤーは早く適正レートへ寄せる
* 試合数の多いプレイヤーは安定させる
* 実装を簡潔に保つ

### 採用案

各プレイヤーの `games_played` に応じて K を以下で決定する。

```text
if games_played < 20:
    K = 40
elif games_played < 100:
    K = 32
else:
    K = 24
```

別表で書くなら以下でもよい。

| 条件                         |  K |
| -------------------------- | -: |
| `games_played < 20`        | 40 |
| `20 <= games_played < 100` | 32 |
| `games_played >= 100`      | 24 |

### 備考

* これはプレイヤーごとに個別適用する
* 同じ試合でも、各プレイヤーの K は異なってよい
* より安定寄りにしたい場合は `32/24/16` などへ調整可能

---

## 試合処理手順

1 試合ごとに以下を実行する。

### 1. 対戦前データ取得

* チーム A の 3 人の `rating`, `games_played`
* チーム B の 3 人の `rating`, `games_played`

### 2. 各プレイヤーの q を計算

```text
q_i = 10^(r_i / 400)
```

### 3. チーム強さを計算

```text
Q_A = sum(q_i for i in A)
Q_B = sum(q_j for j in B)
```

### 4. 期待勝率を計算

```text
p_A = Q_A / (Q_A + Q_B)
```

### 5. 試合結果 y を決定

* A 勝ちなら `y = 1`
* 引き分けなら `y = 0.5`
* B 勝ちなら `y = 0`
* 無効試合なら、レート更新を行わず処理を終了する

### 6. 各プレイヤーの K を決定

`games_played` に応じて K を割り当てる。

### 7. 各プレイヤーの新レートを計算

A 側:

```text
r_i' = r_i + K_i * (q_i / Q_A) * (y - p_A)
```

B 側:

```text
r_j' = r_j - K_j * (q_j / Q_B) * (y - p_A)
```

### 8. 保存

勝ち・負け・引き分けの場合のみ、以下を保存する。

* `rating = r'`
* `games_played += 1`
* `wins` / `losses` / `draws` を結果に応じて更新する

無効試合では、上記の更新を一切行わない。

---

## 実装上の注意

### レートは整数でなくてもよい

内部計算は浮動小数で持ち、表示時だけ丸めるのがよい。

例:

* DB 保存: `1523.4187`
* 表示: `1523`

### q の再利用

同一試合内では、更新前レートから計算した `q` を使い回す。
更新途中で `q` を再計算しない。

### 同時更新する

6 人全員の更新量を先に計算し、その後まとめて反映する。
片側だけ先に更新してから相手側を計算しない。

### 引き分けの扱い

引き分けは `y = 0.5` として処理する。
更新式自体は勝敗時と同一であり、結果値 `y` のみが異なる。

### 無効試合の扱い

無効試合はレート対象外とする。

そのため、無効試合では以下を行わない。

* レート更新
* `games_played` の加算
* `wins` の加算
* `losses` の加算
* `draws` の加算

---

## この方式の利点

* 一軸レートで 3vs3 の期待勝率を自然に計算できる
* 実装が比較的簡単
* Elo に近い説明が可能
* 新規と古参で更新速度を分けられる
* チーム内の強いプレイヤーほど更新寄与が大きい

---

## この方式の限界

* ロール相性を表現できない
* シナジーを表現できない
* 特定の組み合わせだけ強い、という効果を表せない
* 3 人全員の寄与を独立加算的に扱っている

そのため、編成相性が非常に重要なゲームでは限界がある。
ただし Discord コミュニティ向けの初期実装としては十分現実的。
