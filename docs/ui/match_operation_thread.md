# 試合運営 private thread UI 仕様

## UI 識別子

- `ui_type`: `match_operation_thread`

補足:

- この UI は管理者コマンドで常設配置する UI ではなく、`match_id` ごとに動的生成する UI とする。

## 目的

マッチ成立後の参加者向け連絡を、`レート戦マッチング` チャンネル配下の private thread に集約する。

この thread では、少なくとも以下を扱う。

- 試合成立時のチーム分け通知
- 無効試合申請導線
- 親募集と親決定後の進行案内
- 勝敗報告 UI
- 承認対象者向け承認 UI
- 試合結果確定通知
- 観戦応募成功者の合流

## 対象外

- 試合の業務ルールそのもの。
  - [../matches/common.md](../matches/common.md)
- 観戦枠計算や観戦受付条件そのもの。
  - [../matches/spectating.md](../matches/spectating.md)
- slash command の入出力仕様。
  - [../commands/user-commands.md](../commands/user-commands.md)
- 部屋 ID 報告、部屋立て遅延判定、切断判定など、Bot が管理しない運用判断。

## 前提

- Bot は、マッチ成立時に `レート戦マッチング` チャンネル配下へ試合運営用の private thread を 1 つ作成する。
- thread には、少なくとも以下を招待する。
  - 試合参加者全員
  - admin
  - Bot
- 試合参加者 1 人ひとりの在席確認 thread とは別に、この試合運営 thread を持つ。
- `1 match_id` につき、試合運営 thread は 1 つだけ持つ。
- 観戦応募成功者は、観戦受付中である限り、この thread に追加招待される。
- thread の推奨名は [registered_channels.md](registered_channels.md) の命名例に従い、`試合-<match_id>` とする。
- この UI の button は、既存の `/match_*` コマンドの代替導線として扱う。

## thread 作成直後の表示

### 初回メッセージ

- thread 作成直後に、試合成立を知らせるメッセージを送る。
- メッセージには、少なくとも以下を含める。
  - マッチが成立したこと
  - `match_id`
  - 試合形式
  - 試合階級
  - Team A / Team B のチーム分け
  - 試合参加者全員への mention
  - 無効試合とする必要がある場合は `無効試合申請` ボタンを押すこと
- Team A / Team B の表示は、[../matches/common.md](../matches/common.md) と同じチーム区分を使う。
- 初回メッセージには `無効試合申請` button を表示する。

### `無効試合申請` button

- `無効試合申請` button は `/match_void` と同じ業務処理を実行する。
- この button は、試合成立直後から操作できる。
- 参加者は、通常の勝敗報告受付が始まる前でも無効試合を申請できる。
- 通常の勝敗報告受付期間中は、同じ参加者が後から `勝ち`、`負け`、`引き分け` で上書きしてよい。
- 操作できるのは試合参加者本人だけとする。

## 親募集メッセージ

- 初回メッセージの後に、親募集用の追加メッセージを送る。
- メッセージには、少なくとも以下を含める。
  - 部屋立てと試合の進行を行う親を募集すること
  - 親募集期間は 5 分であること
  - 5 分以内に立候補がない場合は、Bot が参加メンバーからランダムに決定すること
- このメッセージには `親に立候補する` button を表示する。

## 自己紹介メッセージ

- 親募集メッセージの後に、自己紹介を促す追加メッセージを送る。
- メッセージには `試合参加者はゲーム内のプレイヤー名を報告してください。` を表示する。

### `親に立候補する` button

- `親に立候補する` button は `/match_parent` と同じ業務処理を実行する。
- 操作できるのは試合参加者本人だけとする。
- 親決定前だけ成功する。
- 親決定後に古い button が押された場合は、`この試合の親はすでに決まっています。` などの業務エラーをそのまま返してよい。

## 親決定後の表示

- 誰かの立候補、または Bot による自動決定で親が決まったら、thread に進行案内メッセージを送る。
- メッセージには、少なくとも以下を含める。
  - 親が誰に決まったか
  - この試合はチーム分け通り、試合形式通りに進行すること
  - 最大 3 試合のマッチであること
  - どちらかのチームが 2 連勝した場合は、コールドとしてその時点でマッチを終了すること

補足:

- Bot はこの 3 試合の進行自体やコールド判定そのものを管理しない。
- ここでの案内は、[../matches/common.md](../matches/common.md) のタイマー仕様に従って勝敗報告時刻を共有するための UI とする。

## 勝敗報告 UI

- `parent_decided_at + 7 分` に到達したら、thread に勝敗報告用メッセージを追加送信する。
- メッセージには、少なくとも以下を含める。
  - 勝敗報告を開始したこと
  - 自分視点で `勝ち`、`負け`、`引き分け` を報告すること
  - 無効試合にすべき場合は `無効試合申請` を押すこと
  - 報告締切時刻
- メッセージには、以下の button を表示する。
  - `勝ち`
  - `負け`
  - `引き分け`
  - `無効試合申請`

### 各 button の対応

- `勝ち` button は `/match_win` と同じ業務処理を実行する。
- `負け` button は `/match_lose` と同じ業務処理を実行する。
- `引き分け` button は `/match_draw` と同じ業務処理を実行する。
- `無効試合申請` button は `/match_void` と同じ業務処理を実行する。
- 操作できるのは試合参加者本人だけとする。
- 勝敗報告受付期間中は、同じ参加者が自分の最新報告を何度でも上書きできる。

## 承認 UI

- 以下のいずれかを満たし、かつ全参加者の報告が全会一致で即時確定しなかった場合、thread に承認メッセージを送る。
  - 全参加者の勝敗報告がそろった
  - 勝敗報告締切に到達した
- メッセージには、少なくとも以下を含める。
  - 仮決定結果
  - 承認が必要であること
  - 承認締切時刻
  - 承認対象者全員への mention 一覧
  - 承認できない場合は証拠を提示したうえで admin へ連絡すること
- 承認 button は、承認対象者が複数いても 1 つだけ表示すればよい。

### `承認` button

- `承認` button は `/match_approve` と同じ業務処理を実行する。
- 同じ button を、承認対象者全員が共用してよい。
- button を押した人がその時点で承認対象者であるかどうかは、業務処理側で判定する。
- 承認対象外の参加者や観戦者が押した場合は、`この試合では承認は不要です。` などの業務エラーを返してよい。

## 結果確定後の表示

- 試合結果が確定したら、thread に結果確定メッセージを送る。
- メッセージには、少なくとも以下を含める。
  - 試合結果が確定したこと
  - 確定結果
- レート更新がある結果では、更新後レートを続けて表示してよい。
- admin が結果を確定または更新した場合は、その旨を表示してよい。
- [../matches/common.md](../matches/common.md) の admin 通知条件を満たす場合は、結果確定後に admin 確認メッセージを同じ thread へ続けて送ってよい。

## 観戦者の追加

- `/match_spectate` または公開の観戦 button で観戦応募が成功した場合、Bot はそのユーザーをこの thread に追加招待する。
- 追加招待は、[../matches/spectating.md](../matches/spectating.md) の観戦受付条件を満たす間だけ行う。
- Bot は必要に応じて、観戦者が追加されたことを thread に通知してよい。

## 可視性と操作権限

- thread を閲覧できるのは、少なくとも以下とする。
  - 試合参加者
  - 観戦応募成功後に追加された観戦者
  - admin
  - Bot
- `親に立候補する`、`勝ち`、`負け`、`引き分け`、`無効試合申請`、`承認` の button は、参加者本人だけが自分の試合操作として使える。
- 観戦者と admin は thread を閲覧できても、参加者向け button 操作で状態変更しない。

## 返答の可視性

- button 押下直後の成功・失敗メッセージは、押した本人にだけ見える返答でよい。
- 一方で、親決定、承認フェーズ開始、結果確定など、試合全体の進行に関わる状態変化は thread へ公開メッセージとして送る。
- 個々の報告ボタン押下のたびに、thread へ公開ログを増やすことは必須としない。

## 冪等性と古い button の扱い

- 同じ thread 内の古い button が残っていても、業務ルール上まだ有効な操作なら使ってよい。
- ただし、状態が進んで無効になった button は押されても状態変更せず、業務エラーだけ返せばよい。
- `1 match_id` に対して thread がすでに存在する場合は、再送や再試行で新しい thread を増やさず既存 thread を再利用する。

## 既存コマンドとの対応

- `親に立候補する` -> `/match_parent`
- `無効試合申請` -> `/match_void`
- `勝ち` -> `/match_win`
- `負け` -> `/match_lose`
- `引き分け` -> `/match_draw`
- `承認` -> `/match_approve`
