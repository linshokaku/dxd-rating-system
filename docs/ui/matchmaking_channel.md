# マッチングチャンネル UI 仕様

## UI 識別子

- `ui_type`: `matchmaking_channel`

## 目的

登録済みユーザーが、`レート戦マッチング` チャンネルに設置された UI からマッチングキューへ参加できるようにする。

## 対象外

- マッチング UI を設置する管理者コマンドの仕様。
- 参加後に利用する在席確認 private thread UI の詳細仕様。
  - [matchmaking_presence_thread.md](matchmaking_presence_thread.md)
- マッチ成立後に作成される private thread の詳細仕様。
  - [match_operation_thread.md](match_operation_thread.md)

## 前提

- この UI は、登録済みユーザー向けの `レート戦マッチング` チャンネルに設置する。
- チャンネル自体の用途と権限は [registered_channels.md](registered_channels.md) を参照する。
- UI 設置コマンドの仕様は [setup_channel.md](setup_channel.md) を参照する。
- この UI は `/join` の代替導線であり、業務処理、参加条件判定、成功文言、失敗文言は `/join` と同じものを使う。
- `/join` は引き続き有効な導線として残す。
- 2 通目の参加状況メッセージ更新は `/update_matchmaking_status` を正規導線とし、`更新する` ボタンは同じ挙動を行う UI 導線として扱う。
- 参加成功後の在席確認とキュー退出は、この公開 UI ではなく、参加時に作成される private thread UI で扱う。
- 詳しい遊び方の案内先は設定 URL `MATCHMAKING_GUIDE_URL` を使う前提とする。

## 公開 UI の表示要素

- 公開 UI は、以下の 5 通の常設メッセージで構成する。
  1. 遊び方メッセージ
  2. 参加状況メッセージ
  3. `1v1` 用の参加 panel
  4. `2v2` 用の参加 panel
  5. `3v3` 用の参加 panel

### 1 通目: 遊び方メッセージ

- 1 通目は説明用メッセージとし、component は付けない。
- 文面は 13 歳でも理解しやすい、短く分かりやすい日本語にする。
- 少なくとも以下を含める。
  - 下の形式別参加 panel から参加したいマッチ形式を選び、その panel で階級を選んでレート戦のマッチングを始められること。
  - マッチしたら、最初にマッチの進行役となる親を決めること。
  - マッチは 3 試合制で、勝った試合が多いチームが勝ちになること。
  - 勝った試合数が同じなら引き分けになること。
  - どちらかのチームが 2 連勝したら、コールドでその時点でマッチを終了すること。
  - マッチが終わったら、`**全員**` が勝敗報告を行うこと。
  - より詳しい遊び方は `MATCHMAKING_GUIDE_URL` のリンク先を案内すること。

### 2 通目: 参加状況メッセージ

- 2 通目は参加状況表示用メッセージとする。
- メッセージ本文には `直近30分の参加状況` のような見出しを含める。
- その下に、全形式 `1v1` / `2v2` / `3v3` の全階級を 1 行ずつ表示する。
- 各行の表示形式は以下とする。
  - `マッチ形式-階級: <max(join - leave - expired, 0)>`
- `join` は、そのキューへの `joined_at` が直近 30 分以内の件数とする。
- `leave` は、そのキューの `status=LEFT` かつ `removed_at` が直近 30 分以内の件数とする。
- `expired` は、そのキューの `status=EXPIRED` かつ `removed_at` が直近 30 分以内の件数とする。
- `matched` は減算しない。
- これは、成立したマッチに含まれるユーザーも直近 30 分の active なユーザーとして残したい意図による。
- 表示順は、マッチ形式と階級定義の表示順に従う。
- メッセージには `更新する` ボタンを 1 つ設置する。
- `/update_matchmaking_status` が、このメッセージ更新の正規仕様とする。
- `更新する` ボタンは `/update_matchmaking_status` 実行時と同じ処理を行う UI 導線とする。
- `/update_matchmaking_status` 実行時、および `更新する` ボタン押下時のどちらでも、公開チャンネルに新しい参加状況メッセージを追加送信しない。

### 3 通目から 5 通目: 形式別参加 panel

- 3 通目から 5 通目は、マッチングキュー参加用の形式別 panel message とする。
- 3 通目は `1v1`、4 通目は `2v2`、5 通目は `3v3` に固定する。
- 各 panel のメッセージ本文は、対象の `match_format` を明示したうえで、在席確認やマッチングキャンセルは参加後に作成される在席確認 private thread で行うことを案内する。
- メッセージ本文では `/present` や `/leave` の slash command を案内しない。
- 各 panel には、以下の component を設置する。
  - 対応する `match_format` 専用の階級プルダウン 1 つ
  - 参加ボタン 1 つ
- 参加ボタン以外に、在席更新ボタンや退出ボタンは置かない。

## 選択肢

### 形式別 panel

- 各 panel は対応する `match_format` に固定される。
- 現時点の panel は以下とする。
  - `1v1`
  - `2v2`
  - `3v3`
- ユーザーは参加したい形式に対応する panel を選んで操作する。

### 階級プルダウン

- `queue_name` を選ぶための component とする。
- 選択肢は、押下している panel に固定された `match_format` に対応する階級定義から組み立てる。
- 初期状態の選択肢は以下とする。
  - `beginner`
  - `regular`
  - `master`
- 初期状態の参加条件は以下とする。
  - `beginner`: 対象フォーマットの現在レートが `1600` 未満
  - `regular`: 対象フォーマットの現在レートに関係なく参加可能
  - `master`: 対象フォーマットの現在レートが `1600` 以上
- 将来フォーマットごとに階級構成が変わった場合は、そのフォーマットで有効な階級だけを表示する。

## 操作フロー

### 参加状況更新コマンドフロー

1. 登録済みユーザーが `/update_matchmaking_status` を実行する。
2. Bot は、同じ guild の `レート戦マッチング` チャンネルに設置された 2 通目の参加状況メッセージを解決する。
3. Bot は、その時点の参加状況を再取得する。
4. Bot は、2 通目のメッセージ本文だけを編集して表示を更新する。
5. Bot は、実行者にだけ見えるテキストメッセージで `参加状況を更新しました。` を返す。

### 参加状況更新ボタンフロー

1. 登録済みユーザーが 2 通目の `更新する` ボタンを押す。
2. Bot は、押下したユーザーにだけ見える ephemeral defer を先に返す。
3. Bot は `/update_matchmaking_status` 実行時と同じ処理を行う。

### キュー参加フロー

1. 登録済みユーザーが、参加したい `match_format` に対応する 3 通目から 5 通目の panel を選ぶ。
2. 登録済みユーザーが、選んだ panel の階級プルダウンで `queue_name` を選ぶ。
3. 登録済みユーザーが、その panel の参加ボタンを押す。
4. Bot は、押下したユーザーにだけ見える ephemeral defer を先に返す。
5. Bot は、押された panel に対応する固定の `match_format` と選択された `queue_name` を使って `/join` と同じ業務処理を実行する。
6. 成功時は、`レート戦マッチング` チャンネル配下に在席確認用の private thread を作成する。
7. 実行結果は、押下したユーザーにだけ見える followup のテキストメッセージで返す。

## 正常時の挙動

- `/update_matchmaking_status` 実行時、および 2 通目の `更新する` ボタン押下時は、その時点の参加状況で 2 通目のメッセージを更新する。
- `/update_matchmaking_status` 成功時は、実行者にだけ見えるテキストメッセージで `参加状況を更新しました。` を返す。
- 2 通目の `更新する` ボタン押下時は、押下直後に ephemeral defer を返し、更新成功時は押下者にだけ見える followup のテキストメッセージで `参加状況を更新しました。` を返す。
- 押された panel に対応する `match_format` と選択された `queue_name` の組み合わせでキュー参加を試みる。
- 参加ボタン押下時は、押下直後に ephemeral defer を返し、最終結果は ephemeral な followup のテキストメッセージで返す。
- 参加成功後は、在席確認と離席を行う private thread を作成し、その thread に初回メッセージと button UI を表示する。
- 成功時の文言は以下とする。
  - `キューに参加しました。5分間マッチングします。`

## エラー時の挙動

- 参加状況更新に失敗:
  - `更新する` ボタンでは ephemeral な followup のテキストメッセージで `参加状況の更新に失敗しました。管理者に確認してください。` を返す。
- 階級が未選択:
  - 参加ボタンでは ephemeral な followup のテキストメッセージで `階級を選択してください。` を返す。
- それ以外の業務エラー:
  - `/join` と同じ失敗文言を返す。
  - 例:
    - `指定したフォーマットは存在しません。`
    - `指定したキューは存在しません。`
    - `現在のレーティングではそのキューに参加できません。`
    - `現在キュー参加を制限されています。`
    - `すでにキュー参加中です。`
    - `キュー参加に失敗しました。管理者に確認してください。`

## 可視性

- 1 通目から 5 通目までの常設メッセージは、対象ユーザーに公開される。
- `/update_matchmaking_status` 実行時、および `更新する` ボタン押下時は、2 通目のメッセージをその場で更新し、公開メッセージを増やさない。
- `/update_matchmaking_status` と `更新する` ボタンの結果メッセージは、成功時も失敗時も実行者または押下者にだけ見えるようにする。
- 参加ボタン押下後の結果メッセージは、成功時も失敗時も押下したユーザーにだけ見えるようにする。
- `レート戦マッチング` チャンネルには、参加結果のたびに公開メッセージを追加送信しない。
- 参加後の在席確認、離席、リマインド、期限切れ通知は private thread 側に集約する。
- 3 通目から 5 通目の形式別参加 panel 自体は、ユーザー操作ごとに更新しなくてよい。

## 冪等性

- 同じユーザーが短時間に `/update_matchmaking_status` を複数回実行しても、最新の参加状況で同じメッセージが再描画されればよい。
- 同じユーザーが短時間に `更新する` ボタンを複数回押しても、`/update_matchmaking_status` と同じ結果で同じメッセージが再描画されればよい。
- 同一ユーザーが短時間に複数回参加ボタンを押した場合でも、キュー参加は 1 回だけ成立すればよい。
- すでに有効な待機キューを持つ状態で押された場合は、重複参加せず `すでにキュー参加中です。` を返す。

## 実装上の読み替え

- この UI は `/join` の入力方法だけを置き換えるものであり、キュー参加の業務ルール自体は変更しない。
- 2 通目の参加状況メッセージ更新は `/update_matchmaking_status` を正とし、`更新する` ボタンはその UI 導線として同じ挙動を行う。
- 3 通目から 5 通目の panel は、それぞれ固定の `match_format` を持つ。
- この UI がユーザー入力として受け取るのは `queue_name` のみとし、`match_format` は押された panel から暗黙に決まる。
- 在席更新とキュー退出は公開チャンネル上では扱わず、参加後に作成される private thread UI に委ねる。
