# マッチング在席確認 thread UI 仕様

## UI 識別子

- `ui_type`: `matchmaking_presence_thread`

## 目的

マッチングキュー参加後の在席報告、離席、残り 1 分リマインド、期限切れ通知を、対象ユーザーが実ユーザーなら本人と admin、ダミーユーザーなら admin が見える private thread に集約する。

## 対象外

- 公開のキュー参加 UI の詳細仕様。
  - [matchmaking_channel.md](matchmaking_channel.md)
- `/present` と `/leave` の slash command 入出力仕様。
  - [../commands/user-commands.md](../commands/user-commands.md)
- マッチ成立後に作成されるマッチ連絡 thread の仕様。

## 前提

- この UI は、通常ユーザーの `/join` 成功時、`/dev_join` 成功時、または公開のキュー参加 UI からの参加成功時に開始する。
- Bot は、登録済みの `レート戦マッチング` チャンネル配下に private thread を 1 つ作成する。
- `/join` と `/dev_join` では、コマンドの実行チャンネルに関わらず同じ `レート戦マッチング` チャンネル配下へ作成する。
- Bot は、対象ユーザーが実ユーザーなら、そのユーザーと admin を thread に参加させる。
- Bot は、対象ユーザーがダミーユーザーなら、admin のみを thread に参加させる。
- thread の閲覧対象は、対象ユーザーが実ユーザーならキュー参加者本人、admin、Bot とし、対象ユーザーがダミーユーザーなら admin と Bot とする。
- Bot が親チャンネルで private thread を作成できることを前提とする。
- `waiting` のキュー行 1 件につき、在席確認 thread は 1 つだけ持つ。
- `left` または `expired` になった後に再参加した場合は、前回 thread を再利用せず、新しい thread を作る。

## thread 作成ルール

- キュー参加の業務処理が成功した後に thread を作成する。
- thread 作成後は、そのキュー参加を表す `waiting` 状態データに対応する `thread_id` を保存する。
- `present`、在席確認リマインド、`leave` では新しい thread を作らない。
- 参加中の導線は、同じ thread に集約する。
- 推奨 thread 名は [registered_channels.md](registered_channels.md) の在席確認 thread 命名規則に従う。

## thread 紐付けによる操作対象の判定

- 在席確認 thread UI の操作対象判定は、現在参加中のキュー状態に保存された `thread_id` を正とする。
- button が押されたときは、押下イベントが発生した thread の `thread_id` を取得する。
- Bot は、対象ユーザーの現在の `waiting` 行に保存された `thread_id` と、押下元 thread の `thread_id` を比較する。
- `thread_id` が一致した場合だけ、その button 操作を現在参加中のキューに対する正当な操作として扱う。
- `thread_id` が一致しない場合は、その thread は現在参加中のキューに紐づいていないものとして扱い、`present` / `leave` の状態変更は行わない。

補足:

- この判定は、同じ thread 内の過去メッセージに残った button を引き続き有効にするために使う。
- いったん `left` または `expired` になって再参加し、新しい thread が作られた後は、古い thread 上の button は `thread_id` 不一致で無効化できる。

## 表示要素

- thread 内では、通常メッセージと button UI を使う。
- button は以下の 2 つを表示する。
  - `在席報告`
  - `離席`
- 初回参加時、在席報告成功時、残り 1 分リマインド時は、同じ 2 ボタンを表示する。
- 離席完了後と期限切れ後は、ボタンを再表示しない。

## 初回表示

- thread 作成直後に、マッチングキュー参加中であることを伝えるメッセージを送る。
- メッセージには、少なくとも以下を含める。
  - 現在マッチングキューに参加中であること。
  - 引き続き参加する場合は `在席報告` ボタンを押すこと。
  - 離席する場合は `離席` ボタンを押すこと。
- 初回メッセージには `在席報告` ボタンと `離席` ボタンを表示する。

## `在席報告` ボタン

### 目的

現在参加中のキューに対して `/present` と同じ在席更新を行う。

### 正常時の挙動

- `在席報告` ボタン押下直後には、押下ユーザーにだけ見える ephemeral defer を返し、成否は followup で返す。
- 先に、押下元 thread の `thread_id` が現在参加中のキュー状態に保存された `thread_id` と一致することを確認する。
- `/present` と同じ業務処理を実行する。
- thread 内に `在席を確認しました。次の期限は5分後です。` を送る。
- 同じメッセージに `在席報告` ボタンと `離席` ボタンを再表示する。

### 失敗時の挙動

- すでにキュー参加中ではない場合:
  - thread 内に `現在このキューには参加していません。再参加する場合は親チャンネルの参加導線から参加してください。` を送る。
- 押下元 thread の `thread_id` が、現在参加中のキュー状態に保存された `thread_id` と一致しない場合:
  - thread 内に `このスレッドは現在参加中のキューには紐づいていません。再参加する場合は親チャンネルの参加ボタンから参加してください。` を送る。
- 内部エラー時の文言は `/present` と同等の失敗文言を使う。

## `離席` ボタン

### 目的

現在参加中のキューに対して `/leave` と同じ離席処理を行う。

### 正常時の挙動

- `離席` ボタン押下直後には、押下ユーザーにだけ見える ephemeral defer を返し、成否は followup で返す。
- 先に、押下元 thread の `thread_id` が現在参加中のキュー状態に保存された `thread_id` と一致することを確認する。
- `/leave` と同じ業務処理を実行する。
- thread 内に `キューから退出しました。` を送る。
- 離席完了後は、同じ thread に `在席報告` ボタンと `離席` ボタンを再表示しない。

### 失敗時の挙動

- すでにキュー参加中ではない場合:
  - thread 内に `現在このキューには参加していません。再参加する場合は親チャンネルの参加導線から参加してください。` を送る。
- 押下元 thread の `thread_id` が、現在参加中のキュー状態に保存された `thread_id` と一致しない場合:
  - thread 内に `このスレッドは現在参加中のキューには紐づいていません。再参加する場合は親チャンネルの参加ボタンから参加してください。` を送る。
- 内部エラー時の文言は `/leave` と同等の失敗文言を使う。

## 在席残り 1 分リマインド

- 在席期限の残り 1 分で、thread 内にリマインドメッセージを送る。
- リマインドメッセージには、少なくとも以下を含める。
  - 在席確認のリマインドであること。
  - 残り 1 分で自動的にキューから外れること。
  - 継続参加するなら `在席報告`、離席するなら `離席` を押すこと。
- リマインドメッセージにも、`在席報告` ボタンと `離席` ボタンを表示する。

## 期限切れによる自動離席

- 在席期限が切れた場合は、thread 内に自動離席したことを報告する。
- 文言には、少なくとも以下を含める。
  - 在席期限切れで自動的にキューから外れたこと。
  - 再参加したい場合は、親チャンネルの参加ボタンから再参加すること。
- 期限切れ後は、同じ thread に `在席報告` ボタンと `離席` ボタンを再表示しない。

## マッチ成立通知

- マッチ成立時は、thread 内に成立通知を送る。
- メッセージには、少なくとも以下を含める。
  - 対象参加者への mention
  - マッチが成立したこと
  - 対応するマッチ運営 private thread へのリンク
- 基本文面は、`<@discord_user_id> マッチ成立です。` と `マッチ運営は <#thread_id> で行ってください。` の 2 行とする。
- この通知に button UI は付けない。

## 可視性と操作権限

- thread 内のメッセージは、キュー参加者本人、admin、Bot に見える。
- `在席報告` ボタンと `離席` ボタンを操作できるのは、対象のキュー参加者本人だけとする。
- admin が thread を閲覧できても、参加者本人以外の button 操作では状態変更しない。
- 本人以外が押した場合の応答は、相手にだけ見えるエラーメッセージで返してよい。
  - 例: `この UI はキュー参加者本人のみ操作できます。`

## 冪等性

- 同じ thread 内の過去メッセージに残った `在席報告` ボタンや `離席` ボタンも、保存済み `thread_id` が一致する限り、現在の `waiting` 行に対する操作として扱ってよい。
- ただし対象キューがすでに `left`、`expired`、`matched` のいずれかなら、状態変更せず案内メッセージだけ返す。
- 再参加によって新しい thread が作成された後は、古い thread 上の button は `thread_id` 不一致として扱い、状態変更しない。
- 同一ユーザーが短時間にボタンを連打しても、業務処理は `/present` と `/leave` の冪等性ルールに従う。
